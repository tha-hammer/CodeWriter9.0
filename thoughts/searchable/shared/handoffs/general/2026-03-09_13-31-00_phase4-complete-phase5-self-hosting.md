## Learnings

### claude-agent-sdk usage
- Package: `pip install claude-agent-sdk` (NOT `claude-code-sdk` — that's the old name)
- Uses global Claude Code auth token — no API keys or env vars needed
- Must `os.environ.pop("CLAUDECODE", None)` before importing to avoid nested session error
- API: `claude_agent_sdk.query(prompt=..., options=ClaudeAgentOptions(...))` returns async iterator of messages
- For pure text generation: `allowed_tools=[]`, `max_turns=1`
- Collect response from `AssistantMessage` → `TextBlock` and `ResultMessage` → `.result`
- stdout from subprocesses spawned by SDK gets captured — use log files for output

### PlusCal label rules for LLM generation
- LLMs consistently miss PlusCal labels inside `either/or` blocks
- The "two-phase action model" requires TWO labels per branch: one for data mutation, one for state transition
- Must include explicit examples of the two-phase pattern in the prompt
- The hand-written `bridge_translator.tla` is a good reference for label placement

### extract_pluscal must handle full modules
- When the LLM outputs a complete TLA+ module (MODULE...====), `extract_pluscal()` must return the FULL module, not strip it to just the algorithm block
- pcal.trans requires MODULE/EXTENDS/CONSTANTS wrapper — bare algorithm blocks fail
- Fixed in `one_shot_loop.py:193-237` by adding a module-detection pattern before the algorithm patterns

### parse_tla VARIABLES regex
- The composer's `parse_tla()` requires a blank line or `\n(` after the VARIABLES declaration
- Test specs must include `\n\n` between VARIABLES and Init/definitions
- Regex: `r'VARIABLES?\s+([\w\s,\\]+?)(?:\n\n|\n\()'`
