# Context for gwt-0029: graceful_shutdown_preserves_progress

IMPORTANT: This spec MUST model COOPERATIVE SIGNAL HANDLING for graceful shutdown.
Do NOT model database schemas, hash comparison, or SDK imports.
The MODULE name MUST be "GracefulShutdown".

## Behavior
A shutdown flag is set by SIGINT. In-flight extractions complete normally.
No new extractions start after the flag is set. Partial progress (already-extracted
records) is preserved in the store.

## Key Variables
- shutdown_requested: boolean (set by signal handler)
- in_flight: set of UUIDs currently being extracted
- completed: set of UUIDs whose extraction finished and was persisted
- started_after_signal: set (should always be empty — invariant)
- pending: set of Tasks to extract
- run_active: boolean

## Key Invariants
- NoNewAfterShutdown: shutdown_requested => started_after_signal = {}
- InFlightCompletes: ~run_active => in_flight = {}
- InFlightDoesNotGrow: shutdown_requested => in_flight \subseteq in_flight_at_signal
- CompletedSubsetTasks: completed \subseteq Tasks
- Partitioned: in_flight, completed, and pending are pairwise disjoint

## PlusCal Constraints — CRITICAL

### One assignment per variable per label
PlusCal forbids assigning the same variable twice within the same label.
Split operations that modify a variable into separate labels.

### Avoid \subseteq on CONSTANTs in invariants
TLC can handle \subseteq when both sides are finite model value sets defined
in the .cfg. Use small finite sets (e.g., Tasks = {t1, t2}).

### Process-local variables
Use `variables` inside `process` for process-local state (like phase, candidate)
rather than global variables. This avoids name collisions in TypeOK.

(e.g., `phase` becomes a variable in the translation). Safer to omit TypeOK
for process-local variables or only check global variables in TypeOK.
