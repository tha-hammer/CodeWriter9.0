# Context for gwt-0078: plan-review-artifacts-gate

## Behavior
Given the artifacts review pass returns verdict 'fail', when `_orchestrate_reviews()` checks the artifacts gate, then no further passes execute; the function returns status='blocked' with blocked_by='artifacts' and only the artifacts result in results.

## Concrete Data Shapes

```
Verdict == {"pass", "fail", "warning", "unknown"}

GateResult == [
    status: {"blocked", "pass", "fail", "warning"},
    blocked_by: STRING | NULL,
    results: DICT(STRING -> PassResult),
    total_cost: REAL
]

PassResult == [name: STRING, verdict: Verdict, cost_usd: REAL]
```

## Key Invariants to Model

- **GateBlocking**: artifacts verdict "fail" => results contains ONLY "artifacts" key, no other pass runs
- **GatePassthrough**: artifacts verdict in {"pass", "warning"} => execution continues to parallel phase
- **BlockedStatus**: status="blocked" if and only if artifacts verdict is "fail"
- **CostAccuracy**: when blocked, total_cost equals only the artifacts pass cost

## Modeling Approach

- CONSTANTS: ArtifactsVerdict (element of Verdict set)
- Variables: gate_checked (BOOLEAN), passes_started (set of names), final_result (GateResult)
- Actions: RunArtifacts, CheckGate, RunParallel (only if gate passes)
- Invariants: GateBlocking, GatePassthrough

## Test Interface (MANDATORY)

```python
import asyncio
from registry.cli import _orchestrate_reviews

# Test gate blocks on fail — mock _run_review_pass to return fail for artifacts
result = asyncio.run(_orchestrate_reviews("self", "plan.md", "/tmp/test", "all", True))
assert result["status"] == "blocked"
assert result["blocked_by"] == "artifacts"
assert list(result["results"].keys()) == ["artifacts"]

# Test gate passes on pass/warning — subsequent passes should appear in results
result = asyncio.run(_orchestrate_reviews("self", "plan.md", "/tmp/test", "all", True))
assert result["status"] != "blocked"
assert "coverage" in result["results"]
```

## Anti-Patterns (DO NOT USE)

- Do NOT model what happens inside the artifacts review pass — just model the verdict
- Do NOT conflate "warning" with "fail" — warning allows continuation
- Do NOT model the gate as a separate component — it's an if-check inside orchestration
