# Context for gwt-0089: WARNING verdict breaks retry loop

## Behavior

Given --auto-fix is active and a review pass returns WARNING, when _run_review_pass() returns to the orchestration loop, then no correction agent is spawned, WARNING is treated as non-blocking, and dependent passes proceed.

## Concrete Data Shapes

```python
# Review pass result with WARNING
pass_result = {
    "name": "coverage",
    "verdict": "WARNING",
    "session_id": "ses-warn01",
    "cost_usd": 0.05,
    "result_length": 2100,
    "result_text": "Minor gaps in edge-case coverage...",
}

# Pass state after WARNING
pass_state = {
    "name": "coverage",
    "attempt": 1,
    "max_retries": 2,
    "verdict": "WARNING",
    "history": [pass_result],
}

# Orchestrator summary with WARNING pass
summary = {
    "status": "warning",          # str: downgraded from "pass" but not "fail"
    "passes": {
        "artifacts": {"verdict": "PASS", ...},
        "coverage": {"verdict": "WARNING", ...},
    },
    "stuck_on": None,             # WARNING does not block
}
```

## Key Invariants to Model

- **NoCorrection_OnWarning**: If verdict == "WARNING", _run_correction_pass() is never called.
- **WarningIsNonBlocking**: WARNING is treated identically to PASS for dependency resolution. Dependent passes proceed.
- **WarningRecorded**: The WARNING verdict is stored; it does not get upgraded to PASS or downgraded to FAIL.
- **LoopTermination**: The retry loop exits on WARNING, same as PASS.
- **SummaryReflects**: Overall summary status is "warning" (not "pass") when any pass has WARNING and none has FAIL.

## Modeling Approach

Model verdict classification as a function mapping verdict to {blocking, non-blocking}. PASS and WARNING both map to non-blocking. Only FAIL maps to blocking. The retry loop checks `isBlocking(verdict)` to decide whether to attempt correction. Model dependency resolution as checking that all upstream passes have non-blocking verdicts.

Key state variables:
- `verdict`: {"PASS", "WARNING", "FAIL", "pending"}
- `isBlocking`: BOOLEAN (derived: verdict == "FAIL")
- `correctionInvoked`: BOOLEAN
- `dependentsEligible`: BOOLEAN

## Anti-Patterns (DO NOT USE)

- Do NOT treat WARNING as blocking. It must not trigger correction or block dependents.
- Do NOT collapse WARNING into PASS. They are distinct verdicts with distinct summary semantics.
- Do NOT model WARNING as requiring special retry logic. It exits the loop the same way PASS does.
- Do NOT use WARNING to mean "retry once then accept." WARNING is immediately accepted.
