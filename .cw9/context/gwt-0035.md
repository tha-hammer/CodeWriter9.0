# Context for gwt-0035: register_idempotent

IMPORTANT: This spec models IDEMPOTENT REGISTRATION via a persistent bindings file. The MODULE name MUST be "RegisterIdempotent".

## Behavior
`_register_payload` uses `.cw9/criterion_bindings.json` as a stable ID allocation map.
For each criterion_id in the input:
- If `"gwt:<criterion_id>"` exists in bindings → reuse the stored gwt_id (no DAG mutation)
- If not → call dag.register_gwt() to allocate a new gwt_id, store in bindings

The bindings file is loaded at the start and saved at the end. Same logic for requirements
using `"req:<cw7_id>"` keys.

Key property: calling register twice with the same criterion_id produces the SAME gwt_id
both times. The DAG does not get duplicate nodes.

## Key Variables
- bindings: function from criterion_id to gwt_id (partial function, initially may be empty)
- dag_nodes: set of gwt_ids in the DAG
- next_id: integer counter for ID allocation
- input_criteria: set of criterion_ids to register (use CONSTANT InputCriteria)
- registered: set of criterion_ids processed so far
- phase: "start" | "processing" | "save" | "done"
- call_count: integer tracking how many times register_gwt was actually called

## Key Invariants
- IdempotentAllocation: \A c \in registered : bindings[c] is the same across all invocations
- NoDuplicateNodes: Cardinality(dag_nodes) = Cardinality(DOMAIN bindings) (one node per binding)
- BindingStable: once bindings[c] = id, it never changes
- SecondCallSkipsAllocate: if c already in DOMAIN bindings at start, call_count does not increment for c
- BoundedExecution: step_count <= MaxSteps

## PlusCal Constraints — CRITICAL

### One assignment per variable per label
PlusCal forbids assigning the same variable twice within the same label.

### Use finite sets in .cfg
InputCriteria = {c1, c2, c3}
MaxSteps = 15

### No duplicate MODULE headers
Exactly ONE `---- MODULE RegisterIdempotent ----` and ONE `====`.

### No \* BEGIN TRANSLATION markers
Do NOT include `\* BEGIN TRANSLATION` / `\* END TRANSLATION` markers.

### CRITICAL: Constant classification hint
For any CONSTANT that is used as a set, add `ASSUME X # {}` after the CONSTANTS block.
This tells the TLC cfg generator to treat it as a finite set, not a numeric value.
For this spec, InputCriteria needs `ASSUME InputCriteria # {}`. MaxSteps is numeric — no ASSUME needed.
