# Context for gwt-0095: Artifacts pass enables parallel dispatch

## Behavior

Given the artifacts pass completes with PASS or WARNING and --auto-fix is active, when the orchestrator schedules the next phase, then coverage and interaction are dispatched concurrently, and each has an independent retry budget.

## Key Invariants to Model

- **ParallelDispatch**: Coverage and interaction are dispatched concurrently after artifacts succeeds. Neither waits for the other.
- **IndependentRetry**: Each pass has its own attempt counter and max_retries budget. Coverage retrying does not affect interaction's budget or scheduling.
- **ArtifactsGate**: Coverage and interaction only dispatch after artifacts completes with a non-blocking verdict (PASS or WARNING).
- **WarningEnablesDispatch**: Artifacts completing with WARNING is sufficient to unblock coverage and interaction. No difference from PASS for scheduling purposes.
- **NoInterference**: If coverage FAILs permanently, interaction continues its own retry cycle unaffected. Only abstraction_gap (which depends on both) is affected.
- **PhaseBarrier**: abstraction_gap only dispatches after BOTH coverage and interaction have resolved (each to PASS, WARNING, FAIL, or blocked).

## Modeling Approach

Model the orchestrator as processing passes in dependency-order phases:

- Phase 1: {artifacts} -- sequential, single pass
- Phase 2: {coverage, interaction} -- concurrent, independent retry loops
- Phase 3: {abstraction_gap} -- waits for phase 2 completion
- Phase 4: {imports} -- waits for phase 3

For phase 2, model two concurrent processes. Each runs its own retry loop. They share no mutable state except the results dict (which they write to disjoint keys). Use asyncio.gather or equivalent to model concurrency.

Key state variables:
- `phase`: Nat (1..4)
- `passBudget[p]`: Nat (independent per pass, initialized to max_retries)
- `passVerdict[p]`: verdict per pass
- `concurrentSet`: set of passes currently executing (|concurrentSet| > 1 in phase 2)

Invariant: `\A p1, p2 \in concurrentSet: dependencies[p1] \cap {p2} = {} /\ dependencies[p2] \cap {p1} = {}`
(concurrent passes never depend on each other)

Invariant: `passBudget[p1]` changes do not affect `passBudget[p2]` for p1 != p2.

## Anti-Patterns (DO NOT USE)

- Do NOT serialize coverage and interaction. They must be concurrent since neither depends on the other.
- Do NOT share retry budgets between concurrent passes. Each pass manages its own attempts independently.
- Do NOT wait for both concurrent passes to finish before starting retries on either. Each pass retries as soon as it fails, without waiting for its sibling.
- Do NOT block coverage dispatch waiting for interaction or vice versa. They are independent after artifacts completes.
- Do NOT treat WARNING on artifacts as blocking. WARNING and PASS are equivalent for dependency resolution.
