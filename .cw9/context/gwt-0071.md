# Context for gwt-0071: observability_template_structure

## Behavior
Given templates/pluscal/observability_state_machine.tla exists with trace_log and audit_log VARIABLES and TraceComplete/AuditComplete invariants,
When a PlusCal algorithm instantiated from this template omits a trace_log append in any action,
Then TLC reports a TraceComplete invariant violation identifying the action that failed to extend trace_log; similarly AuditComplete violations fire for state mutations missing audit_log appends.

## Concrete Data Shapes

The template file adds these to the base state_machine.tla pattern:

```tla+
VARIABLES
    \* ... base state_machine vars ...
    trace_log,        \* Seq of span records: [action |-> String, state |-> String, ts |-> Nat]
    audit_log;        \* Seq of mutation records: [from_state |-> String, to_state |-> String, ts |-> Nat]

define
    \* ... base invariants ...

    \* Every completed action must have a corresponding trace entry
    TraceComplete ==
        step_count > 0 => Len(trace_log) >= step_count

    \* Every state transition must have a corresponding audit entry
    AuditComplete ==
        Len(history) > 0 => Len(audit_log) >= Len(history)
end define;
```

## Key Invariants to Model

- TraceComplete: after N actions, trace_log has at least N entries
- AuditComplete: after N state mutations, audit_log has at least N entries
- TraceLogMonotonic: trace_log never shrinks (Len(trace_log') >= Len(trace_log))
- AuditLogMonotonic: audit_log never shrinks
- BasePreserved: all base state_machine invariants still hold (ValidState, BoundedExecution)

## Modeling Approach

This GWT is unusual — it's verifying a **template**, not an algorithm. The PlusCal spec should model:
- A simple 3-state machine (init → processing → done) with 2 transitions
- Each transition MUST append to trace_log and audit_log
- Invariants TraceComplete and AuditComplete must hold
- Then show that if an action SKIPS the append, TLC catches it

Use a nondeterministic choice: either do the append (correct path) or skip it (violation path).

- CONSTANTS: States = {"init", "processing", "done"}, MaxSteps = 3
- Variables: current_state, step_count, trace_log (Seq), audit_log (Seq), history (Seq), phase
- Actions: TransitionWithTrace (correct), TransitionWithoutTrace (violation demonstrator)
- Invariants: TraceComplete, AuditComplete, ValidState

## Test Interface (MANDATORY)

```python
from pathlib import Path

# Template file existence and structure
template_path = Path("templates/pluscal/observability_state_machine.tla")
assert template_path.exists()

content = template_path.read_text()
assert "trace_log" in content
assert "audit_log" in content
assert "TraceComplete" in content
assert "AuditComplete" in content

# The template should extend base state_machine concepts
assert "current_state" in content
assert "history" in content
assert "step_count" in content
```

## Anti-Patterns (DO NOT USE)

- Do NOT try to model TLC execution itself — model the state machine that the template describes
- Do NOT use complex nested records in TLA+ — keep span/audit records as simple [key |-> value] structs
- Do NOT model file I/O — this is about the TLA+ structure, not reading files
- Do NOT use EXTENDS Sequences unless necessary — bounded sequences via Len() checks work better with TLC
