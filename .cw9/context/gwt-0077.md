# Context for gwt-0077: plan-review-orchestration-flow

## Behavior
Given a set of review passes with dependency constraints (artifacts gates all, coverage and interaction are parallel, abstraction_gap follows parallel, imports is post-only), when `_orchestrate_reviews()` executes with phase='all' and mode='self' and artifacts passes, then artifacts runs first; coverage and interaction run concurrently after artifacts passes; abstraction_gap runs only after both parallel passes complete; imports runs last; total_cost_usd accumulates from all passes.

## Concrete Data Shapes

```
ReviewPass == [name: STRING, prompt_file: STRING, verdict: {"pass", "fail", "warning", "unknown"}, cost: REAL]

OrchestratorState == [
    phase: {"pre", "post", "all"},
    mode: {"self", "external"},
    current_step: {"idle", "artifacts", "parallel", "abstraction_gap", "imports", "done"},
    artifacts_verdict: {"pass", "fail", "warning", "unknown", "not_run"},
    parallel_complete: BOOLEAN,
    results: DICT(STRING -> ReviewPass),
    total_cost: REAL,
    status: {"pass", "fail", "warning", "blocked"}
]
```

## Key Invariants to Model

- **DependencyOrder**: abstraction_gap never starts before ALL parallel passes complete
- **GateEnforcement**: if artifacts verdict is "fail", no other pass ever transitions to running
- **CostMonotonicity**: total_cost only increases (never decreases) as passes complete
- **CompletionConsistency**: when current_step is "done", every applicable pass has a result entry
- **ParallelSafety**: coverage and interaction can be in "running" state simultaneously

## Modeling Approach

- CONSTANTS: ReviewNames (set of pass names), Mode (self/external)
- Variables: state (OrchestratorState), pass_states (function from name to running/done/not_run)
- Actions: StartArtifacts, ArtifactsComplete, StartParallel, ParallelPassComplete, StartAbstractionGap, AbstractionGapComplete, StartImports, ImportsComplete
- Invariants: DependencyOrder, GateEnforcement, CostMonotonicity

## Anti-Patterns (DO NOT USE)

- Do NOT model SDK internals (streaming, message types) — model the orchestration state machine only
- Do NOT model file I/O (reading .md files) — treat prompt loading as atomic
- Do NOT use unbounded integers for cost — use a small finite set of cost values
- Do NOT model concurrent execution as true interleaving — model it as "both started before either completes"
