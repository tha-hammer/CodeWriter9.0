# Context for gwt-0031: semaphore_bounded_concurrency

IMPORTANT: This spec MUST model SEMAPHORE-BOUNDED CONCURRENT EXTRACTION.
Do NOT model hash comparison, DFS traversal, signal handling, or SDK imports.
The MODULE name MUST be "SemaphoreBoundedConcurrency".

## Behavior
The sweep phase uses asyncio.Semaphore(concurrency) to limit the number of
concurrent extract_one calls. At most `concurrency` extractions are active
at any time.

## Key Variables
- concurrency: integer CONSTANT (the semaphore bound, e.g. 2)
- semaphore: integer counter (starts at concurrency, decremented on acquire, incremented on release)
- active: set of UUIDs currently inside the semaphore (holding a slot)
- completed: set of UUIDs that finished extraction
- pending: set of UUIDs waiting to be extracted

## Key Invariants
- SemaphoreNonNeg: semaphore >= 0
- ActiveBounded: Cardinality(active) <= Concurrency
- SemaphoreConsistent: semaphore + Cardinality(active) = Concurrency
- ActiveReleasedOnComplete: completed and active are disjoint
- AllEventuallyProcessed: pending = {} /\ active = {} => completed = Records

## PlusCal Constraints — CRITICAL

### One assignment per variable per label
PlusCal forbids assigning the same variable twice within the same label.
Each label can only assign to a given variable once.

### Model each record as a separate fair process
Use `fair process extractor \in Records` so each record independently
acquires the semaphore, extracts, then releases. This naturally models
concurrency without needing asyncio.gather.

### Use `await` for semaphore acquisition
```
Acquire:
    await semaphore > 0;
    semaphore := semaphore - 1;
    active := active \cup {self};
    pending := pending \ {self};
```
