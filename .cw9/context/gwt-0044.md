# Context for gwt-0044: Crawl bridge creates RESOURCE nodes from crawl.db

## Component
`python/registry/crawl_bridge.py` — `bridge_crawl_to_dag()` Phase 1 (line 31)

## Behavior
Phase 1 of `bridge_crawl_to_dag(crawl_db_path, dag)`:

1. `store.get_all_records()` returns non-external FnRecord objects (WHERE is_external = FALSE)
2. For each record:
   - Construct name: `"ClassName.function_name @ file_path"` or `"function_name @ file_path"`
   - Create Node with `id=rec.uuid`, `kind=NodeKind.RESOURCE`, `name=name`, `description=rec.operational_claim`
   - `dag.add_node(node)` — upserts (overwrites existing node with same ID, no error)
3. `nodes_added` counter increments for every record

## UUID Generation
UUIDs are `uuid5(CRAWL_NAMESPACE, f"{file_path}:{class_name}:{function_name}")` — deterministic and stable across re-ingests. Same function at same path always gets same UUID.

## Key Invariant
- Every non-external record → exactly one RESOURCE node
- `nodes_added == len(get_all_records())`
- Existing nodes with same UUID are silently overwritten (idempotent re-bridge)
- Only `NodeKind.RESOURCE` nodes are created (never BEHAVIOR, REQUIREMENT, etc.)

## State Variables for TLA+ Model
- `records`: set of {uuid, file_path, class_name, function_name, operational_claim}
- `dag_nodes`: map from uuid → Node
- `nodes_added`: counter
- `node_kind`: always RESOURCE for created nodes

## Terminal Condition
After Phase 1, for every record r in records: dag_nodes[r.uuid].kind == RESOURCE. nodes_added == |records|.
