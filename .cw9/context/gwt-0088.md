# Context for gwt-0088: PASS verdict breaks retry loop

## Behavior

Given --auto-fix is active and a review pass returns PASS on any attempt (first or after correction), when _run_review_pass() returns to the orchestration loop, then no correction agent is spawned, the pass is recorded as PASS, and the next eligible pass is scheduled.

## Concrete Data Shapes

```python
# Review pass result dict

    "name": "artifacts",          # str: pass name
    "verdict": "PASS",            # str: one of PASS, WARNING, FAIL
    "session_id": "ses-abc123",   # str: Claude session ID
    "cost_usd": 0.042,            # float: API cost
    "result_length": 1523,        # int: length of result text
    "result_text": "...",         # str: full review output
}

# Orchestrator tracking state per pass
pass_state = {
    "name": "artifacts",
    "attempt": 1,          # int: 1-indexed attempt number (1 = first try)
    "max_retries": 2,      # int: max correction cycles allowed
    "verdict": "PASS",     # str: final verdict after this attempt
    "history": [pass_result],  # list[dict]: all attempt results
}

# Orchestrator summary (relevant fields)
summary = {
    "status": "pass",             # str: overall status
    "passes": {"artifacts": pass_state},
    "stuck_on": None,             # None when nothing blocked
}
```

## Key Invariants to Model

- **NoCorrection_OnPass**: If verdict == "PASS", _run_correction_pass() is never called for that attempt.
- **PassRecorded**: A PASS verdict is stored in the pass history and the pass is marked complete.
- **LoopTermination**: The retry loop for this pass exits immediately on PASS regardless of attempt number.
- **ScheduleNext**: After PASS, the orchestrator evaluates which dependent passes are now eligible and schedules them.

## Modeling Approach

Model the retry loop as a PlusCal while loop with attempt counter. The loop body calls review, checks verdict. On PASS, break immediately without entering the correction branch. Track a `correctionInvoked` flag to assert it remains FALSE when verdict is PASS. Model the scheduling of next passes as a post-loop action that checks dependency satisfaction.

Key state variables:
- `attempt`: Nat (1..max_retries+1)
- `verdict`: {"PASS", "WARNING", "FAIL", "pending"}
- `correctionInvoked`: BOOLEAN
- `passComplete`: BOOLEAN
- `nextScheduled`: set of pass names

## Anti-Patterns (DO NOT USE)

- Do NOT model PASS as triggering a correction pass that happens to be a no-op. PASS must skip the correction branch entirely.
- Do NOT conflate attempt count with retry count. Attempt 1 is the initial run, not a retry.
- Do NOT assume PASS on attempt > 1 is handled differently from PASS on attempt 1. The exit logic is identical.
- Do NOT model the orchestrator as sequential-only; after a PASS, multiple dependents may become eligible concurrently.
