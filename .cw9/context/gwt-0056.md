# Context for gwt-0056: python_profile_condition_compile

## Behavior
Given TLA+ expressions with standard operators, when PythonProfile.compile_condition is called,
a CompiledExpression is returned with valid Python operator mapping.

## Test Interface (MANDATORY — use these exact imports and patterns)

```python
from registry.lang import CompiledExpression, VerifyResult, get_profile, TargetLanguage, PythonProfile
from registry.lang_typescript import TypeScriptProfile
from registry.lang_go import GoProfile
from registry.lang_rust import RustProfile

# Profiles take NO constructor arguments
profile = PythonProfile()       # CORRECT
profile = GoProfile()           # CORRECT

# compile_condition signature:
result: CompiledExpression = profile.compile_condition("x \\in S")

# CompiledExpression fields (these are the ONLY fields):
result.target_expr        # str — the compiled expression
result.original_tla       # str — preserved input
result.variables_used     # list[str] — field names referenced
result.helper_defs        # str — "" for all except GoProfile quantifiers

# compile_assertions signature:
results: dict[str, CompiledExpression] = profile.compile_assertions(verifiers_dict)
```

Anti-Patterns (DO NOT USE — these cause test failures)

- `PythonProfile(some_arg)` ← WRONG. Takes no arguments
- `result.has_error` ← DOES NOT EXIST on CompiledExpression
- `result.get("target_expr")` ← WRONG. It's a dataclass, not a dict
- `result["target_expr"]` ← WRONG. Use attribute access
- Using raw `\E` or `\A` in Python regex patterns ← causes `re.PatternError`
- TLA+ operator names as test variable names (`tla_op`, `go_op`) ← these aren't real API names
