# Context for gwt-0096: abstraction-gap-gating

## Behavior
Given coverage and interaction review passes have both reached terminal verdicts (PASS, WARNING, or FAIL), when `_check_dependencies("abstraction_gap", results, PASS_DEPENDENCIES)` evaluates whether abstraction_gap should be scheduled, then abstraction_gap is scheduled only if both coverage and interaction are PASS or WARNING; if either is FAIL or blocked, abstraction_gap is marked blocked.

## Concrete Data Shapes

```
Verdict == {"pass", "fail", "warning", "blocked"}
PassName == {"artifacts", "coverage", "interaction", "abstraction_gap", "imports"}

PASS_DEPENDENCIES == [
    "artifacts" |-> {},
    "coverage" |-> {"artifacts"},
    "interaction" |-> {"artifacts"},
    "abstraction_gap" |-> {"coverage", "interaction"},
    "imports" |-> {"abstraction_gap"},
]

Results == DICT(PassName -> [verdict: Verdict])

CheckResult == BOOLEAN  \* TRUE = schedule, FALSE = blocked
```

## Key Invariants to Model

- **BothRequired**: abstraction_gap scheduled => coverage.verdict IN {"pass", "warning"} AND interaction.verdict IN {"pass", "warning"}
- **FailBlocks**: coverage.verdict = "fail" OR interaction.verdict = "fail" => abstraction_gap is blocked
- **BlockedPropagates**: coverage.verdict = "blocked" OR interaction.verdict = "blocked" => abstraction_gap is blocked
- **WarningPermits**: coverage.verdict = "warning" AND interaction.verdict = "warning" => abstraction_gap is scheduled
- **MissingBlocks**: if coverage or interaction not yet in results, abstraction_gap is not schedulable

## Modeling Approach

- CONSTANTS: DepVerdicts (coverage verdict, interaction verdict) drawn from Verdict
- Variables: results (map of pass name to verdict), ag_scheduled (BOOLEAN), ag_blocked (BOOLEAN)
- Actions: SetCoverageVerdict(v), SetInteractionVerdict(v), EvaluateAbstractionGap
- Invariants: BothRequired, FailBlocks, BlockedPropagates

## Test Interface (MANDATORY)

```python
from registry.cli import _check_dependencies

PASS_DEPENDENCIES = {
    "artifacts": [],
    "coverage": ["artifacts"],
    "interaction": ["artifacts"],
    "abstraction_gap": ["coverage", "interaction"],
    "imports": ["abstraction_gap"],
}

# Test: both pass -> scheduled
results = {"coverage": {"verdict": "pass"}, "interaction": {"verdict": "pass"}}
assert _check_dependencies("abstraction_gap", results, PASS_DEPENDENCIES) is True

# Test: both warning -> scheduled
results = {"coverage": {"verdict": "warning"}, "interaction": {"verdict": "warning"}}
assert _check_dependencies("abstraction_gap", results, PASS_DEPENDENCIES) is True

# Test: coverage fail -> blocked
results = {"coverage": {"verdict": "fail"}, "interaction": {"verdict": "pass"}}
assert _check_dependencies("abstraction_gap", results, PASS_DEPENDENCIES) is False

# Test: interaction blocked -> blocked
results = {"coverage": {"verdict": "pass"}, "interaction": {"verdict": "blocked"}}
assert _check_dependencies("abstraction_gap", results, PASS_DEPENDENCIES) is False

# Test: coverage missing from results -> not schedulable
results = {"interaction": {"verdict": "pass"}}
assert _check_dependencies("abstraction_gap", results, PASS_DEPENDENCIES) is False
```

## Anti-Patterns (DO NOT USE)

- Do NOT model the internal execution of coverage or interaction -- only their terminal verdicts matter here
- Do NOT treat "unknown" as a terminal verdict for gating purposes -- it means the pass has not completed
- Do NOT conflate "blocked" with "not yet run" -- blocked is a terminal state meaning a dependency failed
- Do NOT model the scheduling mechanism itself -- only model the boolean decision of _check_dependencies
