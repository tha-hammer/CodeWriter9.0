# Context for gwt-0090: FAIL triggers correction

## Behavior

Given --auto-fix is active, a FAIL verdict is returned, and the current attempt is less than max_retries, when _run_review_pass() returns FAIL, then _run_correction_pass() is invoked with extracted findings, a correction agent is spawned, and the review is re-run afterward.

## Concrete Data Shapes

```python
# Review pass result with FAIL
fail_result = {
    "name": "artifacts",
    "verdict": "FAIL",
    "session_id": "ses-fail01",
    "cost_usd": 0.06,
    "result_length": 3200,
    "result_text": "FAIL: Missing API schema for /users endpoint. Missing test for auth flow.",
}

# Extracted findings from result_text
findings = {
    "verdict": "FAIL",
    "issues": [
        "Missing API schema for /users endpoint",
        "Missing test for auth flow",
    ],
    "raw_text": "FAIL: Missing API schema...",
}

# Correction pass result
correction_result = {
    "name": "artifacts_correction",
    "session_id": "ses-corr01",
    "cost_usd": 0.08,
    "result_length": 4500,
    "result_text": "Applied fixes: added schema, added test...",
}

# Pass state mid-retry
pass_state = {
    "name": "artifacts",
    "attempt": 1,            # just failed on attempt 1
    "max_retries": 2,
    "verdict": "FAIL",       # current verdict
    "history": [fail_result],
    "corrections": [correction_result],
}
```

## Key Invariants to Model

- **CorrectionOnFail**: If verdict == "FAIL" AND attempt < max_retries, _run_correction_pass() MUST be called.
- **FindingsExtracted**: _extract_findings() is called on result_text before passing to correction. Correction receives structured findings, not raw text.
- **AttemptGuard**: Correction only fires when attempt < max_retries. At max_retries, no correction.
- **SequentialOrder**: The sequence is always: review -> extract findings -> correction -> review (next attempt). Never correction before review, never two corrections without an intervening review.
- **CorrectionTracked**: Each correction invocation is recorded with its own session_id and cost.

## Modeling Approach

Model the retry loop as:
```
attempt := 1
while attempt <= max_retries:
    verdict := runReview(pass)
    if verdict in {PASS, WARNING}:
        break
    if attempt < max_retries:
        findings := extractFindings(result)
        runCorrection(pass, findings)
    attempt := attempt + 1
```

Key state variables:
- `attempt`: Nat (1..max_retries+1)
- `verdict`: {"PASS", "WARNING", "FAIL", "pending"}
- `correctionCount`: Nat (tracks how many corrections were run)
- `phase`: {"reviewing", "extracting", "correcting", "done"}

The invariant `correctionCount <= attempt - 1` must always hold (can't have more corrections than completed reviews minus one).

## Anti-Patterns (DO NOT USE)

- Do NOT pass raw result_text directly to correction. Always extract findings first via _extract_findings().
- Do NOT run correction when attempt == max_retries. That attempt is the last chance; if it fails, it's permanent.
- Do NOT model correction as happening in parallel with review. They are strictly sequential.
- Do NOT assume correction always fixes the issue. The re-review may still return FAIL.
- Do NOT spawn correction for PASS or WARNING verdicts.
