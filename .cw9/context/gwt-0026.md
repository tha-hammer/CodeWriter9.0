# Context for gwt-0026: staleness_propagation

IMPORTANT: This spec MUST model HASH-BASED STALENESS DETECTION and TRANSITIVE PROPAGATION.
Do NOT model semaphores, workers, rate limiting, concurrent extraction, or API calls.
The behavior is about comparing STORED HASHES vs CURRENT HASHES to detect stale records,
then propagating staleness through a DEPENDENCY GRAPH.

The MODULE name MUST be "StalenessPropagation".

## Behavior
Given record X with changed src_hash and record Y depending on X,
when get_stale_records returns X as directly stale and get_transitive_stale([X.uuid]) is called,
then Y appears in the transitive stale set.

## Two-Phase Staleness Detection

### Phase 1: Direct staleness (get_stale_records)
- Caller provides current_hashes: dict[file_path -> current_hash]
- Query stale_candidates view (all non-external records)
- Compare stored src_hash vs current hash for each file_path
- Return list of directly stale UUIDs

### Phase 2: Transitive propagation (get_transitive_stale)
```sql
WITH RECURSIVE stale(uuid) AS (
    SELECT value FROM json_each(?)          -- seed: directly stale UUIDs
    UNION
    SELECT i.record_uuid                    -- expand: who depends on stale records
    FROM ins i
    JOIN stale s ON i.source_uuid = s.uuid
    WHERE i.source_uuid IS NOT NULL
)
SELECT uuid FROM stale
```

## Direction of Propagation
- ins.record_uuid = the consumer (depends ON source)
- ins.source_uuid = the provider (depended on)
- If source is stale -> consumer is also stale (propagates FORWARD through dependency chain)
- This is the REVERSE direction from get_forward_subgraph (which follows source_uuid FROM a record)

## Key Invariants to Model
- Direct detection: any record whose src_hash differs from current is directly stale
- Transitive closure: if X is stale and Y depends on X, Y is transitively stale
- No false negatives: every downstream dependent of a stale record is included
- Convergence: the recursive CTE terminates (SQLite UNION deduplicates)

## Modeling Approach
- CONSTANTS: Nodes (set of UUIDs), MaxSteps
- Variables: hashes (uuid -> hash), current_hashes (uuid -> hash), edges (set of dependency pairs), direct_stale (set), transitive_stale (set)
- Actions: DetectDirect (compare hashes), PropagateOnce (expand stale set by one hop), Done
- Invariants: DirectCorrect (direct_stale = {n : hashes[n] != current_hashes[n]}), TransitiveComplete (when Done, all reachable dependents included), MonotonicGrowth (stale set never shrinks)
