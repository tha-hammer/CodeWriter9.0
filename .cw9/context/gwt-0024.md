# Context for gwt-0024: upsert_atomicity

IMPORTANT: This spec MUST model the upsert_record operation on a SQLite store.
Do NOT model concurrent workers, extraction counts, or crawl orchestration.
The behavior is about DATABASE TRANSACTION ATOMICITY for a single upsert operation.

## Behavior
Given an existing record with UUID X and content C in crawl.db,
when upsert_record is called with UUID X and updated content C',
then the old record and its ins/outs are replaced atomically.

## Data Model
CrawlStore uses SQLite with WAL journal mode. Three tables involved:
- `records` (uuid TEXT PRIMARY KEY, function_name, class_name, file_path, src_hash, ...)
- `ins` (record_uuid TEXT REFERENCES records ON DELETE CASCADE, name, type_str, source_uuid, ordinal)
- `outs` (record_uuid TEXT REFERENCES records ON DELETE CASCADE, name, type_str, ordinal)

## Implementation Pattern
upsert_record runs:
1. BEGIN IMMEDIATE  (exclusive write lock)
2. UPDATE ins SET source_uuid = NULL WHERE source_uuid = ?  (nullify inbound FK refs from other records)
3. DELETE FROM records WHERE uuid = ?  (CASCADE deletes old ins/outs)
4. INSERT new record + ins + outs
5. COMMIT  (or ROLLBACK on exception)

## Key Invariants to Model
- Atomicity: either all 4 steps complete or none (ROLLBACK on any exception)
- No partial state visible: a concurrent reader (WAL allows concurrent reads) never sees a state where the record exists but ins/outs don't, or vice versa
- Inbound FK preservation: other records' ins rows that pointed to X.uuid via source_uuid are set to NULL (not deleted) — they survive the upsert
- Idempotency: upserting with identical content produces the same final state

## Modeling Approach
Model as a state machine with:
- CONSTANTS: UUIDs (set of record identifiers), MaxSteps
- Variables: records (set of active UUIDs), ins_map (uuid -> set of ins), outs_map (uuid -> set of outs), refs (uuid -> set of UUIDs pointing to it via source_uuid), txn_state (idle/in_progress/committed/rolled_back)
- Actions: BeginUpsert, NullifyRefs, DeleteOld, InsertNew, Commit, Rollback
- Invariants: AtomicityHolds (if txn committed, all 4 mutations visible; if rolled back, none visible), NoOrphanedIns (every ins row belongs to an existing record), RefIntegrity (source_uuid either NULL or points to existing record)
