## Critical Issues

### ŌØī CRITICAL-1: `TestTrace9` is a byte-for-byte duplicate of `TestTrace1` [CORRECTED]

`TestTrace9` (ruby, skeletons=True, web_app) is identical to `TestTrace1`. It contributes zero coverage and signals a copy-paste error. A unique trace is required.

**Fix:** Replace with a combination not covered by any other trace. `rust + skeletons=False + codebase_type="cli"` exercises:
- The non-Python dispatch path with the `skeletons=False` branch
- A codebase type (`cli`) not previously paired with rust

---

### ŌØī CRITICAL-2: `result_empty` is absent from all 10 trace rows [CORRECTED]

The plan states each trace test "asserts every final-state variable," but `result_empty` is not present in the table. This is not a cosmetic gap ŌĆö three invariants (`NonPythonDispatched`, `UnknownSafe`, `MutualExclusion`) directly constrain `result_empty`. Without asserting it, these invariants are never exercised by the trace tests.

**Derivation rules from the invariants:**
| `dispatched_to` | `result_empty` required |
|---|---|
| lang Ōłł NonPythonLangs | `FALSE` (NonPythonDispatched) |
| `"none"` with lang Ōłē KnownLangs | `TRUE` (UnknownSafe) |
| `"python"` | `FALSE` (implied; would otherwise violate PythonPreserved intent) |

---

### ŌØī CRITICAL-3: `TestNonPythonDispatched` omits the `result_empty = FALSE` assertion

The `NonPythonDispatched` invariant is a three-part conjunction:

```
(phase = "finished" Ōł¦ lang Ōłł NonPythonLangs) =>
    (dispatched_to = lang Ōł¦ python_helper = "none" Ōł¦ result_empty = FALSE)
```

The plan describes this class as testing only `python_helper=="none"`. Testing two of three conjuncts while omitting the third means a broken implementation that sets `result_empty = TRUE` for non-Python dispatches would pass all tests despite violating the invariant.

---

### ŌØī CRITICAL-4: `TestUnknownSafe` with "5 other unknown langs" directly violates `TypeInvariant`

The `TypeInvariant` defines the valid domain for `lang` in the final state as `KnownLangs Ōł¬ {"ruby"}`. Ruby is the **only** permitted unknown-language value. Passing five additional unknown strings (e.g., `"cobol"`, `"perl"`) as lang would produce a final state where `lang Ōłē KnownLangs Ōł¬ {"ruby"}`, causing `_verify_type_invariant` to fail before `_verify_unknown_safe` even runs.

This creates a self-defeating test: `_verify_all_invariants` would always error for a reason unrelated to UnknownSafe. The test would be testing the wrong invariant.

**Fix:** `TestUnknownSafe` must use only `"ruby"` as the unknown-lang representative (matching the TypeInvariant domain). To test the robustness of the "skeletons don't override" behaviour, vary the `skeletons` flag and DAG topology with `lang="ruby"`.

---

## Significant Issues

### ŌÜĀ’ĖÅ SIGNIFICANT-1: `TestMutualExclusion` includes `"ruby"` ŌĆö the invariant is vacuously true for it

The `MutualExclusion` invariant is:
```
~(dispatched_to Ōłł NonPythonLangs Ōł¦ result_empty = TRUE)
```

For ruby, `dispatched_to = "none"` (since `"none" Ōłē NonPythonLangs`), so the conjunction is always false regardless of `result_empty`. The invariant is vacuously satisfied ŌĆö no invariant logic is exercised. Including ruby in this test class creates false confidence.

**Fix:** `TestMutualExclusion` must use only `lang Ōłł NonPythonLangs` (go, rust, typescript, javascript). Ruby belongs in `TestUnknownSafe`, not here.

---

### ŌÜĀ’ĖÅ SIGNIFICANT-2: `TestAutodetectBeforeDispatch` is missing three of the seven lang variants

The `AutodetectBeforeDispatch` invariant is unconditional across all languages:
```
(phase Ōłł {dispatching, python_subdispatch, finished}) => autodetect_done = TRUE
```

The plan tests only `python, go, ruby, None`. The langs `rust`, `typescript`, and `javascript` are missing. An implementation that skips setting `autodetect_done = TRUE` during the rust dispatch path would pass all four tested cases and fail silently.

**Fix:** Add `rust`, `typescript`, and `javascript` to this test class to cover all seven lang variants.

---

### ŌÜĀ’ĖÅ SIGNIFICANT-3: `TestCodebaseResolvedBeforeDispatch` ŌĆö "invalid type raises `AssertionError`" is an implementation artifact, not a TLA+ semantic

The TLA+ invariant is:
```
(phase Ōłł {dispatching, python_subdispatch, finished}) => codebase_type Ōłł ValidCodebaseTypes
```

This is a **safety property on reachable states**, not a precondition guard. In TLA+ model checking, an invalid codebase_type reaching the dispatch phase is simply a counterexample ŌĆö the spec doesn't prescribe *how* the implementation prevents it.

Testing for a Python `AssertionError` couples the test to one specific implementation choice. A compliant implementation could instead return early with an empty result set, raise a different exception, or reject the input at a validation layer. The test should verify the invariant's semantic intent: **the system never reaches `phase="finished"` with `codebase_type Ōłē ValidCodebaseTypes` after having dispatched**. If the implementation raises, that is an acceptable mechanism ŌĆö but the test assertion should be `with pytest.raises((AssertionError, ValueError))` or, better, verify the invariant post-call.

---

### ŌÜĀ’ĖÅ SIGNIFICANT-4: `phase` and `autodetect_done` are absent from the trace table

The plan states each test "asserts every final-state variable." The final state has at least: `phase`, `lang`, `dispatched_to`, `python_helper`, `result_empty`, `autodetect_done`, `step_count`. The table currently specifies only `dispatched_to` and `python_helper`. Omitting `phase`, `autodetect_done`, and `step_count` from the trace table leaves the trace tests under-specified.

---

## Minor Issues

### Ōä╣’ĖÅ MINOR-1: `step_count` assertions are absent from trace tests

`BoundedExecution` requires `step_count Ōēż MaxSteps`. While `_verify_all_invariants` covers the Ōēż4 bound, each trace test should additionally assert an exact `step_count` value (derivable from the number of TLA+ actions taken in that trace). This makes regressions where the implementation takes a wrong number of steps detectable.

---

### Ōä╣’ĖÅ MINOR-2: `TestEdgeCases` ŌĆö `None`Ōåö`"python"` equivalence test is underspecified

The plan mentions testing "`None`Ōåö`"python"` equivalence" but does not specify what "equivalence" means semantically. If it means `discover_entry_points(dag, None, ...)` and `discover_entry_points(dag, "python", ...)` return identical `(entry_points, state)` tuples, that is a valid and testable property. This must be made explicit; otherwise the test has no falsifiable assertion.
