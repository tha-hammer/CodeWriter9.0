## Review Findings

Four semantic defects were found. Here is the analysis, followed by the corrected test plan.

---

### Defect 1 â€” `test_json_object_not_array` is misclassified (semantic error)

**Location:** `TestInvariantMalformedImpliesError`

**Problem:** `MalformedImpliesError` fires on `~json_valid` (the bytes cannot be parsed as JSON at all). A top-level JSON object `{"resource_type": â€¦}` **is** valid JSON â€” `json_valid = TRUE`. It fails because it violates the schema (`~all_fields_present`), so it belongs under `InvalidSchemaImpliesError`. Placing it in `MalformedImpliesError` verifies the wrong invariant antecedent, meaning a loader that distinguishes parse errors from schema errors would satisfy both invariants correctly yet still fail this test, or vice-versa.

**Fix:** Move the test to `TestInvariantInvalidSchemaImpliesError`.

---

### Defect 2 â€” `TestInvariantPhaseValid` never asserts that an error was actually raised (missing assertion)

**Location:** `TestInvariantPhaseValid.test_only_file_not_found_or_value_error_on_failure`

**Problem:** The test only checks that *if* an exception is raised it is one of the two legal types. If `load_cross_cutting_rules` silently returns (e.g., returns `[]`) for a malformed file or missing file, the `try/except` block takes neither branch and the test passes without complaint. PhaseValid requires the call to *terminate* with either a return or a raise; this test fails to verify the raise side.

**Fix:** Track whether each call raised, and `pytest.fail` if it did not for inputs that must raise.

---

### Defect 3 â€” `assert rules == [] or rules is None` weakens NoPartialResults and SafeResult

**Location:** `assert_no_partial_results`, `assert_safe_result`

**Problem:** The TLA+ invariant is `error /= "" => result = {}`. The Python equivalent is `result == []`. Accepting `None` permits a loader that returns `None` on error to pass both invariants, which is not conformant with the spec (the return type is always `list`). In all existing tests `rules` is initialised to `[]` and the `is None` branch is dead â€” but it is a latent hole that would allow a regression to slip through.

**Fix:** Use `assert rules == []` only.

---

### Defect 4 â€” Dead statement inside `with pytest.raises` block (misleading code)

**Location:** `test_trace1_missing_fields_raises_value_error`

**Problem:** `error_raised = True` is placed *inside* the `with pytest.raises(ValueError):` block, after the call site. Because `pytest.raises` requires the exception to be raised, that line is unreachable. The comment tries to explain this, but the statement should simply be removed; its presence implies intent that isn't there.

**Fix:** Remove the dead assignment.

**Problem:** The test only checks that *if* an exception is raised it is one of the two legal types. If `load_cross_cutting_rules` silently returns (e.g., returns `[]`) for a malformed file or missing file, the `try/except` block takes neither branch and the test passes without complaint. PhaseValid requires the call to *terminate* with either a return or a raise; this test fails to verify the raise side.

**Fix:** Track whether each call raised, and `pytest.fail` if it did not for inputs that must raise.

---

### Defect 3 â€” `assert rules == [] or rules is None` weakens NoPartialResults and SafeResult

**Location:** `assert_no_partial_results`, `assert_safe_result`

**Problem:** The TLA+ invariant is `error /= "" => result = {}`. The Python equivalent is `result == []`. Accepting `None` permits a loader that returns `None` on error to pass both invariants, which is not conformant with the spec (the return type is always `list`). In all existing tests `rules` is initialised to `[]` and the `is None` branch is dead â€” but it is a latent hole that would allow a regression to slip through.

**Fix:** Use `assert rules == []` only.

---

### Defect 4 â€” Dead statement inside `with pytest.raises` block (misleading code)

**Location:** `test_trace1_missing_fields_raises_value_error`

**Problem:** `error_raised = True` is placed *inside* the `with pytest.raises(ValueError):` block, after the call site. Because `pytest.raises` requires the exception to be raised, that line is unreachable. The comment tries to explain this, but the statement should simply be removed; its presence implies intent that isn't there.

**Fix:** Remove the dead assignment.
